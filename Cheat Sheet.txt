The Big Question: Why do we use branching if cloning + pulling does the same job?
If your entire team just clones the repo and pushes directly to main (solving conflicts on their laptops via git pull), you are technically merging. But here is why teams don't do it:

Reason A: You avoid the "Race to Push" Panic
Imagine a team of 5 researchers working on a machine learning project.

You spend 4 hours writing code and try to push. Rejected! Someone else pushed first.

You run git pull, spend 15 minutes fixing their conflicts, and try to push again. Rejected again! A third teammate pushed while you were fixing the conflicts.

You get stuck in an endless loop of pulling, fixing conflicts, and praying your push goes through before someone else beats you to it.

With Branching: You create a branch called hosein/model-fix. You can push your branch to GitHub anytime you want without pulling anyone else's code first. Your branch sits safely online in its own lane. No one can block you.

Reason B: Peer Review & Code Quality (Pull Requests)
If you push directly to main, your code goes live instantly. If you accidentally left a typo that crashes the training script, the whole team is blocked until you fix it.

With Branching: 1. You finish your work on your separate branch and push it online.
2. You open a Pull Request on GitHub. This is a webpage that says: "Hey team, I want to merge my branch into main. Please look at my code changes."
3. Your advisor or teammate clicks the link, sees a clean git diff of your work, types comments like "Looks great, but change learning_rate to 0.01", and hits Approve.
4. Only after the approval is the branch safely merged into main.

Reason C: Keeping main Working at All Times
If an admissions committee or a collaborator visits your GitHub link right now, they expect to see code that works. If you work directly on main, the code is constantly in a half-broken state while you are actively working on it. Branching ensures that main is a pristine, flawless zone, while the messy, unfinished work happens safely off to the side.

Restoring (restore and restore --staged ) 

simple words : undo uncommitted changes

>> git restore <filename> / restore . --
 
When to use it: You modified a file, completely messed up the code, it doesn't compile, and you just want to wipe out your recent edits and start over from your last saved state.


2- >> git restore --staged <flename> / restore --staged . --

It un-stages a file. It removes the file from the Staging Area, but leaves your actual text changes inside your Working Directory completely safe.

When to use it: You accidentally typed git add . or git add deep_learning_model.py before it was ready, and you want to pull it out of the staging room so it doesn't get included in the next commit.





git restore --staged <file>`**: **True.** You use this when you want to pull a file *back* out of the Staging Area room because you aren't ready to commit it yet. It leaves your actual code edits completely untouched in your working directory.

git restore <file>`**: **True.** You use this when you want to throw away your recent messy edits in your Working Directory and get back the clean "original file" (the last saved version from your staging area or last commit).


2.`git reset` vs `git restore --staged**`

Your statement that **`git reset` does the same job as `git restore --staged` is historically correct!

If you type:

```bash
git reset HEAD file.txt

```

It does the **exact same thing** as:

```bash
git restore --staged file.txt

```

- Stash -

Use when You are in the middle of writing a complex new feature or refactoring a file. The code is completely broken, half-written, and won't compile. Suddenly, your teammate or advisor messages you: "Hey, there is a critical bug in the main branch, I need you to fix it right now!"

You can't commit your broken, half-written code (commits should always be clean and working).

You can't switch branches or pull updates because Git will block you or overwrite your messy workspace files.

The Solution: You stash your broken work to clear your desk, fix the urgent bug, and then un-stash your work to resume exactly where you left off.

 example:

1- >> git stash --

now you switch using >> git checkout branchname

then after that you come back to your branch you were working with

now if you see your files are not included you changes you added 

simply you can get them back with :

2- >> git stash pop -- 


* multiple stash : Git packs up your current workspace, puts it into a new box, and throws it onto the top of the pile.

Because it acts like a stack, it follows a rule called LIFO (Last In, First Out). This means when you run git stash pop, Git always grabs the very top box (the newest one you stashed) first.

When you have multiple items on your shelf, Git labels them using an index system that looks like this:

stash@{0}: The top of the pile (Newest stash)

stash@{1}: The item below it

stash@{2}: The item below that (Oldest stash)

The Multi-Stash Toolbox

3- >> git stash -m "my message" -- 

: Highly recommended when doing multiple stashes! It lets you put a label on the box so you don't forget what's inside.

4- >> git stash list -- 

Peeks at your shelf and shows you your pile of boxes with their labels.

git stash pop: Grabs the top box (stash@{0}), brings it back to your code, and throws the empty box away.

5- >> git stash apply stash@{1} --

: Brilliantly brings back a specific box from the middle of the pile without deleting it from the shelf.



After adding the stash one and commiting is not stash anymore-


6- >> git stash apply (if you have one stash list it works )

when use git apply when git pop :

git stash pop: Throws the changes back into your files and deletes the box from the stash shelf.

git stash apply: Copies the changes back into your files but leaves the box sitting safely on the stash shelf.

 
merge VS Apply

Use git merge when a feature is 100% complete, fully tested, and you want to permanently combine its entire history into another main branch.

Use git stash apply when you just want to copy-paste a quick, uncommitted block of helper code or configuration settings across your workspace without making it a permanent historical event yet.

 - Revert -

git revert is Git’s official "public safety eraser." It is the safest, cleanest way to undo a mistake that has already been pushed to GitHub because it fixes the error without destroying or erasing your project's historical timeline.

Instead of traveling back in time to delete a bad commit, git revert looks at the bad commit, calculates the exact opposite of what you did, and automatically creates a brand new commit that rolls back those changes.


The Core Difference: git revert vs git reset:

While both commands undo things, they are used in completely opposite scenarios in the real world. The golden rule boils down to: Is your mistake public (on GitHub) or private (only on your laptop)

summary Rule of Thumb

If you made a messy commit on your laptop and haven't pushed it online yet, use git reset to wipe your tracks clean.

If your commit is already sitting on GitHub, never use reset. Use git revert to cleanly step forward out of the mistake.

 - REBASE - 

git rebase is Git's ultimate time-travel tool for rewiring history. It allows you to take all the new commits you wrote on a side branch, lift them up, and slide them cleanly onto the very end of another branch (like main).

The Merge Way (The Historical Blend)
When you run git merge main, Git combines the two branches by creating a brand-new Merge Commit at the end.

The Result: Your history timeline splits and comes back together like a knot. It preserves exactly how and when things happened historically.

The Rebase Way (The Perfect Line)
When you run git rebase main on your side branch, Git temporarily detaches your commits, downloads the new commits from main, and then re-plays your commits one by one on top of the fresh updates.

The Result: It rewrites your history to look like a perfectly straight line, as if you started working after your teammate pushed their changes!


Local/Private branch? Use git rebase as much as you want to keep your personal timeline clean before sending it to the team.

Public/Shared branch? Never rebase. Use standard git merge so everyone’s timelines stay aligned and peaceful!

- Pull Request - 

Pull Request (PR) is not actually a Git command—it is a feature invented by platforms like GitHub, GitLab, and Bitbucket.

In simple terms, a Pull Request is a formal "Hey team, I finished my work, please review it and merge it!" notification. It is a dedicated webpage where your team can look at your code changes, discuss them, run automated tests, and safely merge your side branch into the main production branch.




 
Summary Checklist of a PR Workflow

1- Create a branch on your laptop.

2- Write code and commit your changes locally.

3- Push the branch up to GitHub (git push origin your-branch).

4- Open the PR on the GitHub website.

5- Discuss, fix, and review the code with your team.

6- Merge the PR to combine it into the main branch.
